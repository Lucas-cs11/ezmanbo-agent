\section{四级容错需求解析链}

需求解析模块负责将工程师输入的自然语言选型需求转换为 \texttt{RequirementC\\onstraints} 结构化约束对象。其核心难点在于，工程需求文本通常具有表述形式多样、参数边界隐含和领域知识依赖强等特点。为提升解析结果的稳定性与可控性，系统设计了四级串联容错策略。其中，前级负责完成基础语义映射，后级针对已知解析缺陷进行局部修正，而非对前级结果进行整体重写，从而在保留 LLM 语义理解能力的同时增强规则约束下的工程可靠性，具体容错机制如下：

\textbf{第一级——LLM语义解析}：系统调用 LLM 模型，通过精心设计的提示词从输入文本中提取 \texttt{category}、\texttt{topology}、输入/输出电压与电流、温度范围及等级偏好等字段，并以 JSON 格式返回。

\textbf{第二级——规则化覆盖}：通过关键词匹配规则对 LLM 输出进行修正。例如，“降压”“buck”“DC-DC”等词触发 \texttt{dc\_dc\_converter} 类别，“升压”“boost”触发对应拓扑。同时，对否定句式进行专项处理，如“非车规”“不需要车规”被正确映射为工业级而非触发车规过滤。

\textbf{第三级——电压方向纠正}：当器件类别为 DC-DC 转换器且输入输出电压均已提取时，系统根据输入电压与输出电压的大小关系强制修正拓扑判断：若输入电压大于输出电压，则将拓扑设置为 \texttt{buck}；若输入电压小于输出电压，则将拓扑设置为 \texttt{boost}。

\textbf{第四级——温度范围正则匹配}：通过多格式正则表达式提取“-40到85°C”“-40 to 125°C”“-40\textasciitilde{}85”等写法中的温度边界值以处理 LLM 未正确解析的温度区间。电流提取优先匹配 mA 单位并除以 1000 转换为 A，从而避免如将“输出 500 mA”被误识别为 500 A 的量级错误。


\section{eZ-PLM API 集成与多前缀分组查询}

\subsection{HMAC-SHA256 签名认证}

eZ-PLM API 采用基于 HMAC-SHA256 的签名认证机制。每次请求需在头部携带 API 密钥、Unix 时间戳、UUID 随机数和签名值。签名内容由请求方法、请求路径、规范化查询参数、时间戳和随机数组成，并按固定顺序拼接形成待签名字符串。随后，客户端以 API 密钥作为 HMAC 种子，对该字符串执行 HMAC-SHA256 计算，并将结果进行 Base64URLSafe 编码以生成最终签名。为降低重放攻击风险，客户端在每次请求时均生成新的时间戳和随机数，从而保证签名内容的唯一性。

\subsection{多前缀分组查询策略}

由于 eZ-PLM API 主要支持基于制造商型号前缀的关键字检索，而不支持直接按照器件类别或拓扑关键词查询，系统构建了从类别与拓扑到制造商型号段的映射机制。如表 \ref{tab:keywords} 所示，Buck 拓扑映射至 \texttt{TPS54}、\texttt{TPS62}、\texttt{LM2596}、\texttt{ADP23} 等前缀，Boost 拓扑映射至 \texttt{TPS61}、\texttt{TPS63}、\texttt{MCP1640} 等前缀，LDO 器件则映射至 \texttt{TPS79}、\texttt{MCP1703}、\texttt{ADP312} 等前缀。系统针对每个前缀独立发起查询，每次最多返回 50 条结果，并以制造商型号为键进行去重，最终总召回上限设定为 200 条。

针对 API 返回结果中输出电压字段可能同时包含固定输出值与最大电压范围的问题，系统进一步设计了基于 MPN 的电压推断方法。该方法通过正则表达式从型号字符串中提取固定输出电压信息，例如从 \texttt{LM2596S-5.0} 中识别出 5.0 V 输出，并在器件匹配与评分阶段优先采用该推断值。由此可减少固定输出电压与电压范围字段混用导致的过滤误差。

数据检索采用“API 优先、离线数据库兜底”的双路径策略。当 API 密钥可用时，系统优先通过多前缀分组查询获取实时在产物料；当 API 无返回结果或密钥不可用时，则自动回退至离线器件数据库。候选器件随后经过五维过滤，包括类别匹配、输入电压覆盖、输出电压 $\pm 5\%$ 容差、输出电流下限和温度范围/等级匹配，以保留满足全部约束的器件。


\section{双模自适应混合评分引擎}

评分引擎是 Pipeline 推理链中将候选器件列表转化为排序推荐结果的核心模块。其总体架构——包括双模自适应切换逻辑、纯规则模式与 LLM 增强模式的权重分配公式、以及去重与推荐分级策略——已在第二章第 2.3 节中完成方案论证，此处不再赘述。本节聚焦于评分引擎内部各子维度的具体计算逻辑、LLM 软评分的提示词工程细节，以及评分流程中涉及的数值边界处理机制。

\subsection{参数匹配评分的逐维计算}

如式（\ref{eq:param}）所示，参数匹配分 $S_{\text{param}}$ 是基于实际存在的约束项数量 $N$ 进行动态归一化后得出的综合指标。其下属四个子维度各自遵循独立的打分函数。

输入电压覆盖维度考察器件的额定输入电压范围是否包含用户指定的标称输入电压。当器件的 $V_{\text{in,min}} \leq V_{\text{in,nominal}} \leq V_{\text{in,max}}$ 成立时，该维度得分为 1.0，否则得分为 0。这一二元判定逻辑将输入电压视为器件可用性的硬性前提——一旦不满足，该器件在工程上即不可被纳入考虑。

输出电压匹配维度针对固定输出电压器件设计了分级评分策略。系统首先通过 MPN 推断函数从器件型号中提取固定输出电压值（如 LM2596S-5.0 被推断为 5.0 V），随后与用户需求的输出电压进行比较：若二者偏差小于 0.01 V，视为精确匹配，得分为 1.0；若偏差不超过需求值的 5\%，视为可接受的近距匹配，得分为 0.6；偏差超过 5\% 时，得分降至 0.1。对于输出电压为可调范围（ADJ）的器件，该维度不参与评分，$N$ 值相应减 1，以避免对可调器件产生不公平的惩罚。

输出电流满足维度采用余量比例折算法。当器件的最大输出电流 $I_{\text{out,part}}$ 不低于需求值 $I_{\text{out,demand}}$ 时，得分按照 $s_{\text{iout}} = \min(1.0,\; I_{\text{out,part}} / (2 \times I_{\text{out,demand}}))$ 计算，即器件电流能力达到需求值两倍时获得满分，刚好满足需求时得 0.5 分。这种设计既保证了基本需求的满足，又通过余量系数对超额能力给予正向激励。

温度范围覆盖维度同样采用二元判定：当器件的工作温度区间完全覆盖用户指定的温度范围（即 $T_{\text{min,part}} \leq T_{\text{min,demand}}$ 且 $T_{\text{max,part}} \geq T_{\text{max,demand}}$）时，得分为 1.0，否则为 0。当用户未指定温度约束时，该维度不参与评分。

\subsection{供应链稳定性评分的分段计算}

供应链稳定性分 $S_{\text{supply}}$ 基于两个可量化指标进行分段映射。库存水平采用四级阶梯函数：超过 1000 件对应 100 分，100 至 1000 件对应 70 分，1 至 100 件对应 40 分，零库存则直接降至 10 分。生命周期状态作为截断因子叠加于库存得分之上：一旦器件的生命周期状态被标记为 \texttt{obsolete}（已停产）或 \texttt{discontinued}（已停止供货），无论其库存得分如何，$S_{\text{supply}}$ 的上限被强制截断至 10 分，以充分反映选用停产器件所伴随的长期供应中断风险。

\subsection{LLM 软评分的提示词工程与安全边界}

当系统进入 LLM 增强模式时，\texttt{score\_part\_with\_llm()} 函数负责构造两段提示词并调用 LLM 进行评估。系统提示词将 LLM 的角色定位为"资深电子工程师"，并要求其严格基于所提供的参考设计内容进行评分，不得凭空推断；用户提示词则包含三部分信息——用户的原始选型需求文本、待评估器件的完整电气参数（型号、制造商、输入电压范围、输出电流、温度范围等），以及从 eZ-PLM 参考设计接口获取的最多三条真实设计案例（每条案例取其名称与前 300 字符的描述文本）。LLM 被要求以如下 JSON 格式返回评分结果：

\begin{lstlisting}[caption={LLM 评分返回格式},label={lst:llm_fmt}]
{"application_score": 85, "design_risk_score": 78, "reasoning": "一句话说明"}
\end{lstlisting}

系统对 LLM 返回的原始文本执行两阶段安全处理。第一阶段为 JSON 提取：在返回内容中定位第一个 \texttt{\{} 和最后一个 \texttt{\}}，截取中间的 JSON 片段进行解析，以避免 LLM 在 JSON 前后附加解释性文字导致的解析失败。第二阶段为数值边界处理：对 \texttt{application\_score} 和 \texttt{design\_risk\_score} 两项分值执行 $\max(0, \min(100, v))$ 的 clamp 操作，将越界值强制裁剪至合法区间。若 JSON 解析失败或任一字段缺失，函数返回空字典，触发 Pipeline 调度层向纯规则模式的自动回退。这种"乐观调用、保守校验"的设计原则使得 LLM 的输出始终处于规则引擎的安全边界之内。

\subsection{核心型号提取与去重合并}

eZ-PLM API 返回的同一型号族常包含多个仅在封装后缀或卷带标识上存在差异的变体。为消除此类冗余对排序结果的干扰，系统在评分完成后执行核心型号提取与去重合并。\texttt{\_extract\_core\_mpn()} 函数通过正则表达式去除 MPN 字符串中的质量等级后缀（如 /NOPB、/LF03）、卷带标识（如 -TR、-T\&R）及末尾的非功能性封装变体字符，从而提取出代表该器件电气型号的核心标识（如从 LM2596S-5.0/NOPB 提取出 LM2596S-5.0）。随后，以核心 MPN 为键对全部候选器件进行分组，每组仅保留综合得分最高的变体，并将被合并变体的完整型号信息以"同类变体：LM2596SX-5.0/NOPB（SX 封装）"的格式追加至保留记录的 reasons 字段，确保变体信息的可追溯性。最终按照第二章第 2.3.4 节所述的 Top-N 分级策略完成排名标注与列表截断。

\section{双层风险评估架构}

风险报告生成采用规则验证与LLM叙述分离的两阶段设计。

第一阶段由规则引擎对供应链和工程两大类风险进行确定性检测。供应链风险涵盖无候选器件（数据覆盖缺口）、推荐器件不足（全部为备选或不推荐）、单一供应商集中度过高、库存水平异常（零库存或低于100件）、生命周期处于停产状态等情形；工程风险则关注车规需求未满足、参数匹配度过低、关键数据字段缺失（无数据手册URL）等条目。每条检测结果均按HIGH、MEDIUM、LOW三档标注严重程度，并附带缓解建议，生成结构化RiskItem列表。

第二阶段把已验证的RiskItem列表传入LLM，仅允许其对现有风险条目撰写工程背景阐述，不赋予其独立生成新风险结论的权限。LLM输出的每个叙述段落均被显式标注数据来源（"规则引擎检测结果"），从而把风险事实和语言表达完全分离开来。报告整体结构遵循ISO 31000\cite{iso31000}规范——文件信息、执行摘要、评估方法说明、5$\times$5风险矩阵\cite{iec60812}、供应链风险详述、工程技术风险详述、人工复核清单和结论建议。

\section{检索增强生成（RAG）工程知识库}

RAG模块基于ChromaDB持久化向量存储与all-MiniLM-L6-v2嵌入模型（384维）构建\cite{lewis2020rag}，当前包含29条工程知识条目，覆盖Buck/Boost/LDO设计公式、USB-C PD功率需求、热管理、车规AEC-Q100认证要求\cite{aecq100}、PCB Layout规范\cite{ipc2221,ipc2152}、供应链评估框架及EMI/可靠性指南共11个类别。Pipeline调度层在每次选型分析时自动根据用户需求与解析约束构建查询文本，检索Top-5相关条目，并将检索结果以EvidenceIR形式注入证据链——三份输出报告（BOM、风险评估、拓扑分析）均会在独立节中展示RAG检索到的工程知识、相关度评分及来源引用。
