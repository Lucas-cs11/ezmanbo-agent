\section{四级容错需求解析链}

需求解析模块负责将工程师输入的自然语言选型需求转换为 \texttt{RequirementC\\onstraints} 结构化约束对象。其核心难点在于，工程需求文本通常具有表述形式多样、参数边界隐含和领域知识依赖强等特点。为提升解析结果的稳定性与可控性，系统设计了四级串联容错策略。其中，前级负责完成基础语义映射，后级针对已知解析缺陷进行局部修正，而非对前级结果进行整体重写，从而在保留 LLM 语义理解能力的同时增强规则约束下的工程可靠性，具体容错机制如下：

\textbf{第一级——LLM语义解析}：系统调用 LLM 模型，通过精心设计的提示词从输入文本中提取 \texttt{category}、\texttt{topology}、输入/输出电压与电流、温度范围及等级偏好等字段，并以 JSON 格式返回。

\textbf{第二级——规则化覆盖}：通过关键词匹配规则对 LLM 输出进行修正。例如，“降压”“buck”“DC-DC”等词触发 \texttt{dc\_dc\_converter} 类别，“升压”“boost”触发对应拓扑。同时，对否定句式进行专项处理，如“非车规”“不需要车规”被正确映射为工业级而非触发车规过滤。

\textbf{第三级——电压方向纠正}：当器件类别为 DC-DC 转换器且输入输出电压均已提取时，系统根据输入电压与输出电压的大小关系强制修正拓扑判断：若输入电压大于输出电压，则将拓扑设置为 \texttt{buck}；若输入电压小于输出电压，则将拓扑设置为 \texttt{boost}。

\textbf{第四级——温度范围正则匹配}：通过多格式正则表达式提取“-40到85°C”“-40 to 125°C”“-40\textasciitilde{}85”等写法中的温度边界值以处理 LLM 未正确解析的温度区间。电流提取优先匹配 mA 单位并除以 1000 转换为 A，从而避免如将“输出 500 mA”被误识别为 500 A 的量级错误。


\section{eZ-PLM API 集成与多前缀分组查询}

\subsection{HMAC-SHA256 签名认证}

eZ-PLM API 采用基于 HMAC-SHA256 的签名认证机制。每次请求需在头部携带 API 密钥、Unix 时间戳、UUID 随机数和签名值。签名内容由请求方法、请求路径、规范化查询参数、时间戳和随机数组成，并按固定顺序拼接形成待签名字符串。随后，客户端以 API 密钥作为 HMAC 种子，对该字符串执行 HMAC-SHA256 计算，并将结果进行 Base64URLSafe 编码以生成最终签名。为降低重放攻击风险，客户端在每次请求时均生成新的时间戳和随机数，从而保证签名内容的唯一性。

\subsection{多前缀分组查询策略}

由于 eZ-PLM API 主要支持基于制造商型号前缀的关键字检索，而不支持直接按照器件类别或拓扑关键词查询，系统构建了从类别与拓扑到制造商型号段的映射机制。如表 \ref{tab:keywords} 所示，Buck 拓扑映射至 \texttt{TPS54}、\texttt{TPS62}、\texttt{LM2596}、\texttt{ADP23} 等前缀，Boost 拓扑映射至 \texttt{TPS61}、\texttt{TPS63}、\texttt{MCP1640} 等前缀，LDO 器件则映射至 \texttt{TPS79}、\texttt{MCP1703}、\texttt{ADP312} 等前缀。系统针对每个前缀独立发起查询，每次最多返回 50 条结果，并以制造商型号为键进行去重，最终总召回上限设定为 200 条。

针对 API 返回结果中输出电压字段可能同时包含固定输出值与最大电压范围的问题，系统进一步设计了基于 MPN 的电压推断方法。该方法通过正则表达式从型号字符串中提取固定输出电压信息，例如从 \texttt{LM2596S-5.0} 中识别出 5.0 V 输出，并在器件匹配与评分阶段优先采用该推断值。由此可减少固定输出电压与电压范围字段混用导致的过滤误差。

数据检索采用“API 优先、离线数据库兜底”的双路径策略。当 API 密钥可用时，系统优先通过多前缀分组查询获取实时在产物料；当 API 无返回结果或密钥不可用时，则自动回退至离线器件数据库。候选器件随后经过五维过滤，包括类别匹配、输入电压覆盖、输出电压 $\pm 5\%$ 容差、输出电流下限和温度范围/等级匹配，以保留满足全部约束的器件。


\section{双模自适应混合评分引擎}

评分引擎是 Pipeline 推理链中将候选器件列表转化为排序推荐结果的核心模块。其总体架构——包括双模自适应切换逻辑、纯规则模式与 LLM 增强模式的权重分配公式，以及去重与推荐分级策略——已在第 2.3 节完成方案论证。本节重点介绍评分引擎内部各子维度的计算逻辑、LLM 软评分提示词设计及数值边界处理机制。

\subsection{参数匹配评分的逐维计算}

参数匹配分 \texttt{S\_param} 基于实际存在的约束项数量 $N$ 进行动态归一化，包含四个子维度：

\begin{itemize}
\item \textbf{输入电压覆盖}：当器件输入电压范围包含用户标称输入电压，即 $V_{\text{in,min}} \le V_{\text{in,nominal}} \le V_{\text{in,max}}$，得分为 1，否则为 0。该二元判定确保输入电压作为硬约束的有效性。

\item \textbf{输出电压匹配}：对于固定输出电压器件，系统通过 MPN 推断函数提取额定输出电压值，并与用户需求比较：偏差小于 0.01 V 得 1.0 分，偏差不超过 5\% 得 0.6 分，超过 5\% 得 0.1 分；可调输出（ADJ）器件不计入 $N$。

\item \textbf{输出电流满足}：采用余量比例折算法，得分计算为
\begin{equation}
    s_{\text{iout}} = \min\bigl(1.0, I_{\text{out,part}} / (2 \times I_{\text{out,demand}})\bigr)
\end{equation}
当器件电流达到需求值两倍时满分 1.0，恰好满足需求时得 0.5 分。

\item \textbf{温度范围覆盖}：当器件工作温度区间完全覆盖用户需求区间，即 $T_{\text{min,part}} \le T_{\text{min,demand}}$ 且 $T_{\text{max,part}} \ge T_{\text{max,demand}}$，得分为 1，否则为 0。若用户未指定温度约束，则该维度不计入评分。
\end{itemize}

\subsection{供应链稳定性评分}

供应链稳定性分 \texttt{S\_supply} 基于库存水平与生命周期状态进行分段计算。库存水平按四级阶梯映射：$>1000$ 件为 100 分，100–1000 件为 70 分，1–100 件为 40 分，0 件为 10 分。若器件生命周期状态为 \texttt{obsolete}（已淘汰）或 \texttt{discontinued}（已停供），库存得分上限强制截断为 10 分，以反映长期供应中断风险。

\subsection{LLM 软评分提示词与安全边界}

在 LLM 增强模式下，\texttt{score\_part\_with\_llm()} 构造系统提示词（将 LLM 角色定位为“资深电子工程师”）与用户提示词（包含原始需求文本、器件完整电气参数及最多三条参考设计案例摘要），并要求 LLM 严格基于提供内容返回 JSON 格式评分：
\begin{lstlisting}[caption={LLM 评分返回格式},label={lst:llm_fmt}]
{"application_score": 85, "design_risk_score": 78, "reasoning": "一句话说明"}
\end{lstlisting}

系统对 LLM 返回进行双阶段安全处理：首先截取首尾大括号内 JSON 内容解析，避免附加文本导致解析失败；其次对 \texttt{application\_score} 和 \texttt{design\_risk\_s\\core} 执行 $\max(0, \min(100, v))$ clamp，越界值裁剪至合法区间。若解析失败或字段缺失，则返回空字典，触发 Pipeline 回退至纯规则模式。

\section{双层风险评估架构}

风险报告生成采用规则验证与 LLM 叙述分离的双层设计。

\textbf{第一阶段——规则引擎检测}：规则引擎对供应链与工程技术两类风险进行确定性检测。供应链风险包括无候选器件（数据覆盖缺口）、推荐器件数量不足（仅备选或不推荐）、单一供应商集中度过高、库存异常（零库存或低于100件）、生命周期处于停产状态等；工程风险包括车规需求未满足、参数匹配度低、关键字段缺失（如无数据手册 URL）等。每条检测结果均按照高（HIGH）、中（MEDIUM）、低（LOW）三级标注严重程度，并附缓解建议，形成结构化 \texttt{RiskItem} 列表。

\textbf{第二阶段——LLM 背景说明}：已验证的 \texttt{RiskItem} 列表传入 LLM，仅允许生成针对现有风险条目的工程背景说明，不可独立产生新风险结论。LLM 输出的每段文本均显式标注数据来源为“规则引擎检测结果”，确保风险事实与语言表述完全分离。整体报告结构遵循 ISO 31000 \cite{iso31000}，包含文件信息、执行摘要、评估方法说明、5$\times$5 风险矩阵 \cite{iec60812}、供应链风险详述、工程技术风险详述、人工复核清单及结论建议。

\section{检索增强生成（RAG）工程知识库}

RAG 模块基于 ChromaDB 持久化向量存储及 \texttt{all-MiniLM-L6-v2} 嵌入模型（384 维）构建 \cite{lewis2020rag}，当前包含 29 条工程知识条目，覆盖 Buck/Boost/LDO 设计公式、USB-C PD 功率需求、热管理、车规 AEC-Q100 认证要求 \cite{aecq100}、PCB Layout 规范 \cite{ipc2221,ipc2152}、供应链评估框架及 EMI/可靠性指南等 11 个类别。Pipeline 调度层在每次选型分析时根据用户需求及解析约束构建查询文本，检索 Top-5 相关条目，并将检索结果以 \texttt{EvidenceIR} 形式注入证据链。三份输出报告（BOM、风险评估、拓扑分析）均在独立节中展示 RAG 检索到的工程知识、相关度评分及来源引用。
