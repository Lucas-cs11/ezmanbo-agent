\section{Pipeline 调度实现}

Pipeline 的主入口 \texttt{analyze()} 函数定义于 \texttt{agent\_orchestrator.py}，负责将解析、检索、评分、证据构建与报告生成五个阶段串联为端到端执行链路。首先，\texttt{parse\_requirement()} 将用户输入转换为 \texttt{RequirementConstraints} 对象；随后以该对象为输入调用 \texttt{search\_parts()} 获取候选器件列表。检索阶段完成后，调度层执行两项增强操作：  

\begin{enumerate}
\item 若环境变量配置了 LLM 密钥，则对排名前 10 且来源为 \texttt{ezplm} 的候选器件逐一调用 \texttt{fetch\_reference\_designs()}，通过 eZ-PLM 参考设计接口获取每个器件最多三条设计案例，构建 \texttt{ref\_designs\_map} 字典供评分引擎使用。
\item 调用 \texttt{\_query\_rag\_knowledge()} 自动触发 RAG 知识库检索，根据用户需求及解析后的约束字段（拓扑、输出功率、等级偏好等）构建查询文本，调用 ChromaDB 语义检索接口获取 Top-5 相关条目，并将其作为 \texttt{rag\_knowledge} 类型 \texttt{EvidenceIR} 注入证据链。
\end{enumerate}

本地处理环节耗时控制在 20 毫秒以内，主要开销集中于 eZ-PLM API 网络请求和 LLM 远程调用，保证了高效的数据流与实时性。

\section{需求解析模块}

\texttt{requirement\_parser.py} 的唯一公开接口为 \texttt{parse\_requirement(text: str) -> RequirementConstraints}。除了第三章 3.1 节中算法原理的介绍，本节从软件实现角度补充两个关键细节：

\begin{itemize}
\item \textbf{LLM 结果归一化}：解析模块使用 \texttt{\_norm\_cat()} 和 \texttt{\_norm\_topo()} 将 LLM 输出的多样化表述统一为标准值。例如，将 \texttt{"DC-DC"、"buck"、"降压"、"电源管理"} 均归一化为 \texttt{dc\_dc\_converter}，将 \texttt{"降压"、"buck"、"buck converter"} 归一化为 \texttt{buck}，避免后续匹配失效。
\item \textbf{规则覆盖策略}：规则引擎仅在已验证的缺陷场景下对 LLM 输出进行定向修正。关键词匹配规则仅在出现 \texttt{"降压"、"buck"、"升压"、"boost"、"ldo"、"线性稳压"} 时触发；电压比较兜底仅在类别为 DC-DC 且输入输出电压均提取时生效；mA 单位前置匹配与否定句式处理仅在匹配到特定正则模式时执行。该策略保留 LLM 对多样化自然语言的理解优势，同时保证关键参数的工程可靠性。
\end{itemize}

\section{器件检索模块}

\texttt{ezplm\_client.py} 实现了“API 优先、离线数据库兜底”的双路径策略，核心接口为 \texttt{search\_parts(constraints: RequirementConstraints) -> List[Par\\tIR]}。模块会遍历 \texttt{\_API\_KEYWORDS} 映射表，将类别/拓扑组合映射至制造商型号前缀（如 Buck 对应 \texttt{TPS54}、\texttt{TPS62}、\texttt{LM2596}、\texttt{ADP23} 等），针对每个前缀独立调用 \texttt{\_request\_json()} 发起 HMAC-SHA256 签名 GET 请求，每前缀最多获取 50 条结果，总召回上限为 200 条。返回的 JSON 经 \texttt{\_map\_api\_part\_to\_partir()} 转换为 \texttt{PartIR} 对象，再通过 \texttt{\_part\_matches\_constraints()} 执行五维过滤（类别匹配、输入电压覆盖、输出电压 ±5\% 容差、输出电流下限、温度范围与等级匹配），最终生成候选器件列表。

\section{混合评分与 LLM 调用模块}

混合评分引擎（\texttt{scoring.py}）的核心接口为 
\texttt{score\_candidates(constraints, candidates, ref\_designs\_map=None) -> List[ScoredPart]}。该函数依次调用 \texttt{\_compute\_param\_score()} 和 \texttt{\_compute\_supply\_score()} 计算规则维度基础得分及评分原因列表，并根据 \texttt{ref\_designs\_map} 可用性与 LLM 密钥状态选择采用纯规则模式或 LLM 增强模式。两种模式的加权公式分别见式（\ref{eq:rule}）和式（\ref{eq:hybrid}），子维度打分函数详见第 3.3 节。评分完成后，\texttt{\_extract\_core\_mpn()} 对同系列变体执行去重合并，并按 Top-N 策略完成排名和列表截断。

LLM 调用模块（\texttt{llm\_client.py}）屏蔽了底层模型差异。其 \texttt{call\_openai\_chat\\(messages, model=None, temperature=0.0) -> str} 函数通过读取 \texttt{OPENAI\_BAS\\E\_URL} 和 \texttt{OPENAI\_API\_KEY} 动态适配不同 OpenAI 兼容端点，实现 DeepSeek、GPT、Claude 等模型的无缝切换。模块提供两个专用函数：\texttt{parse\_requirement\_wi\\th\_llm()} 用于需求解析，\texttt{score\_part\_with\_llm()} 用于 LLM 增强评分，提示词模板内嵌函数体，通过系统提示词设定角色边界，用户提示词传入结构化上下文。

\section{证据构建模块}

证据构建模块（\texttt{evidence.py}）公开接口为 
\texttt{build\_evidence(scored\_parts, constraints) -> List[EvidenceIR]}。该函数遍历评分结果，为每个候选器件生成八类证据条目：输入电压覆盖、输出电流满足、温度范围覆盖、库存水平（库存为零时触发 \texttt{need\_human\_review=True}）、生命周期状态（停产或停止供货时标记人工复核）、车规认证（需求车规但器件非车规时置信度设为 0.30 并标记人工复核）、国产化（标注制造商是否国产）、数据手册可用性（无手册链接时置信度设为 0.50 并标记人工复核）。

\section{报告生成模块}

报告生成模块由 \texttt{output\_bom.py} 与 \texttt{output\_generator.py} 协同实现，用于将评分结果、证据链和风险条目转换为标准化工程文档。其中，\texttt{output\_bom.py} 中的 \texttt{generate\_bom()} 负责生成符合 IPC 命名习惯的 BOM 选型清单。该文档包括推荐器件表、备选器件表和关键参数对比矩阵，并在末尾附加评分方法说明，以增强报告的自解释性。

\texttt{output\_generator.py} 主要负责风险评估报告、拓扑分析报告及结构化拓扑数据的生成。其中，\texttt{generate\_risk\_report()} 将规则引擎生成的 \texttt{RiskItem} 列表展开为符合 ISO 31000 风险管理框架的 Markdown 文档，内容包括执行摘要、评估方法说明、$5\times 5$ 风险矩阵、供应链风险详述、工程技术风险详述、人工复核清单和结论建议。每条风险均附带数据来源声明，以区分规则验证得到的风险事实与 LLM 生成的工程叙述。

\section{ReAct Agent 交互模块}

Agent 交互模块由 \texttt{agent\_tools.py} 和 \texttt{react\_agent.py} 协同实现。其中，\texttt{agent\_tools.py} 基于 LangChain 的 \texttt{@tool} 装饰器定义了四个专用工具：\texttt{search\_\\components} 封装完整的选型分析 Pipeline，用于返回候选器件列表与评分摘要；\texttt{query\_design\_knowledge} 封装 RAG 知识库检索，用于返回相关工程知识条目；\texttt{find\_alternative\_parts} 封装替代器件查找逻辑，用于返回兼容替代列表与对比摘要；\texttt{generate\_full\_report} 触发完整报告生成流程，用于返回各类工程文档的文件路径。每个工具均包含自然语言描述和参数 Schema，使 Agent 能够在推理过程中根据任务需求自主选择调用。

\texttt{react\_agent.py} 中的 \texttt{create\_component\_agent()} 函数通过 LangChain 的 \texttt{create\_agent()} API 将上述工具注册到 ReAct Agent 实例，并通过系统提示词约束工具调用边界。具体而言，系统限制搜索操作至多执行一次、知识检索至多执行一次、总工具调用次数不超过五次，并明确禁止编造不存在的器件型号，以降低无效搜索循环和型号幻觉风险。\texttt{run\_agent(user\_input, session\_id)} 函数负责多轮对话上下文管理，通过全局会话字典维护各会话的 \texttt{HumanMessage}、\texttt{AIMessage} 与 \texttt{ToolMessage} 历史记录；后续请求只需传入相同的 \texttt{session\_id}，即可恢复对应会话上下文。同时，系统将递归深度限制为 \texttt{recursion\_limit=8}，以保证响应时延可控。

此外，\texttt{\_validate\_part\_numbers\_in\_response()} 函数在 Agent 生成最终回复前，对文本中出现的器件型号进行数据库交叉校验。若某型号既不符合已知制造商前缀白名单，也不存在于离线器件数据库中，系统将在回复末尾追加风险提示，从而进一步缓解 LLM 型号幻觉问题。

\section{Web 服务与前端展示}

Web 服务层（\texttt{main.py}）基于 FastAPI 框架如~\ref{sec:architectrue}所述向外暴露了五个核心服务端口, 同时通过 \texttt{RequestValidationError} 异常处理器统一处理请求体格式校验失败和 UTF-8 中文编码兼容问题，保障了前端与推理层之间通信的稳定性。

前端展示层（\texttt{frontend/streamlit\_app.py}）基于 Streamlit 框架构建了可视化的选型交互界面。用户通过左侧边栏的快捷预设或自由文本输入框提交自然语言需求后，前端以工具调用步骤卡片的形式实时展示 Pipeline 各阶段的执行进度和中间结果；候选器件的评分结果被呈现为按推荐/备选/不推荐分级的可折叠仪表盘面板，每张卡片展示综合得分进度条、五维子评分项、关键电气参数速览和评分原因标签；证据链与参数对比模块被整合为一个增强面板——顶部以四项统计指标（证据总条数、覆盖器件数、证据类型数、高置信度条目数）汇总证据概览，中部支持按器件型号筛选查看具体证据条目，底部以表格形式展示全部候选器件的关键电气参数（输入电压范围、最大输出电流、温度范围、车规标识、生命周期、库存、价格及综合得分），方便工程师快速进行横向参数对比。
