\section{研究背景与问题定义}
电子元器件选型在硬件产品研发全生命周期中具有基础性和关键性作用。以 DC-DC 电源模块设计为例，德州仪器（Texas Instruments）、亚德诺半导体（Analog Devices）、意法半导体（STMicroelectronics）、微芯科技（Microchip Technology）等数十家制造商提供了数万种相关型号，工程师通常需要逐项核查并交叉比对其输入输出电压范围、最大输出电流、开关频率、工作温度等级、封装约束、库存水平和生命周期状态等多维参数；同时，还需查阅数据手册以验证外围元件的设计公式，并综合评估供应链稳定性 \cite{pecht2008parts}。该流程通常耗时数小时以上，且主要存在三方面瓶颈：第一，工程知识检索高度碎片化，设计规范分散于数据手册、应用笔记和工程师个人经验中；第二，供应链风险缺乏系统化的量化评估手段 \cite{chen2021supplyrisk}；第三，选型结论和输出文档格式不够规范，难以直接支撑工程评审或产品生命周期管理平台的集成应用。

针对上述问题，若能充分利用元器件管理平台的结构化物料数据与参考设计资源，并结合智能化推理技术，有望提升选型过程的自动化程度、风险识别能力和文档生成规范性。硬禾科技旗下的 eZ-PLM 是面向电子元器件全生命周期管理的开放平台（www.ezplm.cn/www.ezmpl.com），涵盖超过 110 万条在产物料数据与参考设计资源，并支持进行实时器件查询和设计案例检索，为器件检索、参考设计复用和供应链信息获取提供了数据基础。在此基础上，近年来大语言模型（LLM）与智能体（Agent）技术在工程决策辅助领域也展现出较强潜力 \cite{xi2023rise,li2023llmagent}，其自然语言理解、任务分解与交互推理能力可与领域规则引擎形成互补。因此，将 eZ-PLM 的结构化物料数据与 LLM/Agent 的智能推理能力相结合，有望将传统人工选型流程转化为智能化、可追溯的自动化决策过程。

\section{现有方案的不足}

现有的元器件选型辅助方案大致可分为两类：一类是基于传统工业软件的 PLM/ERP 方案，另一类是基于通用大语言模型（Large Language Model, LLM）的问答式方案。前者具备真实物料数据和结构化管理能力，但智能化程度有限；后者具备较强的自然语言理解和知识表达能力，但在数据真实性、结果可追溯性和工程可靠性方面仍存在明显不足。

\subsection{传统 PLM/ERP 方案的局限}

以 ERP 和 PLM 平台为代表的传统工业软件，如 SAP Integrated Product Development、Oracle NetSuite、PTC Windchill 等，是当前企业进行元器件信息管理和产品生命周期管理的重要工具。这类平台通常能够提供基于表单的参数过滤、库存查询、型号管理和 BOM 导出等基础功能，但在面向复杂工程选型任务时，仍存在以下局限。

\textbf{（1）缺乏自然语言交互能力。} 工程师通常需要将设计需求手动映射为平台中的筛选字段和参数范围，难以直接通过自然语言表达选型意图。例如，对于“需要一个 12 V 转 5 V、3 A 的车规级降压方案”这一需求，工程师需要将其拆解为“类别 = DC-DC 转换器”“拓扑 = Buck”“输入电压约为 12 V”“输出电压 = 5 V”“输出电流 $\geq$ 3 A”“温度等级 = 车规”等多个筛选条件，并逐项填入平台表单。该过程不仅耗时，也容易因人为疏漏导致约束条件遗漏或表达不一致。

\textbf{（2）缺乏应用场景适配度评估能力。} 传统平台通常依据参数范围返回符合条件的器件列表，但难以进一步判断器件与具体应用场景之间的匹配程度。例如，两款器件均满足“输入电压 4.5--40 V、输出 5 V/3 A”的基本要求，其中一款可能面向通信设备并强调低噪声特性，另一款则可能面向工业电机驱动并强调宽温和抗干扰能力。传统平台通常无法对这类应用场景差异进行智能评估，工程师仍需依赖数据手册和个人经验完成最终判断。

\textbf{（3）供应链风险评估能力薄弱。} 多数 PLM/ERP 平台主要展示库存数量、生命周期状态和供应商信息等静态数据，缺乏对供应商集中度 \cite{pecht2002electronic}、交期波动、地缘政治风险和停产风险等多维因素的统一量化评估与分级预警能力 \cite{goel2021supply}。尤其对于器件停产（End of Life, EOL）或过时化（Obsolescence）风险，现有平台往往难以及时给出早期预警和主动替代推荐 \cite{sandborn2013forecasting}。

\subsection{通用 LLM 方案的幻觉与不可溯源性}

与传统 PLM/ERP 方案相比，以 ChatGPT、Claude、DeepSeek 等为代表的通用 LLM 在自然语言理解、任务解释和工程知识表达方面具有明显优势。然而，元器件选型属于高可靠性工程决策场景，直接依赖通用 LLM 进行器件推荐仍面临“幻觉”（hallucination）与不可溯源问题 \cite{wang2023survey}。在选型任务中，该问题主要表现为以下三种形式。

\textbf{（1）型号编造。} 当用户询问某款电源模块是否适配特定输入/输出条件时，模型可能生成看似合理但实际并不存在的器件型号，并附带虚构的电压、电流、封装或认证信息。例如，在未接入真实器件数据库的情况下，模型可能输出类似“TPS12345-Q1，4.5--36 V 输入，5 V/3 A 输出，AEC-Q100 认证”的推荐结果。该类结果在语言形式上具有较强迷惑性，但缺乏真实物料数据支撑。

\textbf{（2）参数捏造或误引。} 即使模型给出了真实存在的器件型号，也可能错误描述其关键参数。例如，模型可能将 LM2596S-5.0 的输出电流误报为 5 A，或错误声称某器件支持 AEC-Q100 认证。由于输入电压、输出电流、温度等级和认证状态等参数直接影响器件可用性，此类错误可能导致严重的工程选型偏差。

\textbf{（3）结果不可溯源。} 通用 LLM 的回答通常难以明确追溯到具体数据来源，无法说明推荐结论来自哪一份制造商数据手册、应用笔记、参考设计，或哪一次实时 API 查询结果。因此，工程师难以验证推荐结果的真实性和时效性，也难以满足工程评审和质量管理流程对设计决策可追溯性的要求。

\subsection{两类方案之间的能力断层}

综上，当前元器件选型辅助方案存在明显的能力断层：传统 PLM/ERP 方案具备真实物料数据和结构化参数管理能力，但缺乏自然语言需求理解、应用适配度评估和供应链风险分析能力；通用 LLM 方案具备自然语言交互和知识表达能力，但缺乏真实数据约束、事实验证机制和结果可追溯性。如何将 eZ-PLM 平台中的真实物料数据和参考设计资源，与 LLM 的智能推理能力相结合，并通过工程规则引擎对关键事实进行约束与校验，是提升元器件选型自动化、可靠性和可追溯性的关键\cite{lewis2020rag,gao2023retrieval}。

\section{本文贡献}

本文面向 eZ-PLM 平台，设计并实现了一套融合 LLM 与工程规则引擎协同推理的智能元器件选型与风险评估 Agent。该系统将自然语言需求输入、器件检索、混合评分、风险评估与工程文档生成等环节串联为端到端自动化流程，可输出BOM选型清单、供应链风险评估报告和电路拓扑分析报告三类标准化工程文档。系统的主要技术贡献体现在以下五个方面。

\textbf{（一）提出五阶段线性流水线（Pipeline）架构}。该架构将“自然语言需求解析—实时 API 器件检索—混合智能评分—证据链构建—标准化报告生成”五个环节组织为可自动执行的处理链路，每一阶段的输出均作为下一阶段的输入，从而实现从用户意图到工程文档的自动转换。在此基础上，系统进一步集成基于 LangChain 框架的推理行动 ReAct 智能体交互模式 \cite{yao2022react}，支持在多轮对话中维护上下文，并自主编排搜索、知识检索、替代查找和报告生成等工具的调用顺序。其中，Pipeline 模式保障处理效率和输出一致性，Agent 模式提升复杂需求和追问场景下的交互推理能力，二者共同构成批量处理与交互式选型兼容的执行机制。

\textbf{(二) 设计了四级容错需求解析链，以提升自然语言需求到结构化约束字段的转换可靠性}。第一级由 LLM 完成语义解析，将“宽压在 4.5 到 36 V 之间”“需要输出五伏三安的降压芯片”等多样化表述映射为结构化约束；第二级通过规则化关键词匹配对类别和拓扑进行修正；第三级引入电压比较兜底规则，在输入电压与输出电压均已知时根据物理转换方向锁定拓扑类型，即 $V_{\mathrm{in}} > V_{\mathrm{out}}$ 时设为 \texttt{buck}，$V_{\mathrm{in}} < V_{\mathrm{out}}$ 时设为 \texttt{boost}，以避免将“5 V 转 3.3 V”等典型降压需求误判为 LDO；第四级通过多格式温度范围正则匹配，将“-40 到 85~\textdegree C”“-40 to 125~\textdegree C”“-40\textasciitilde85~\textdegree C”等表达统一解析为温度上下限。此外，该解析链还引入 mA 单位前置匹配和否定句式专项处理，可将“输出 500 mA”正确转换为 0.5 A，并将“非车规”识别为否定语境，从而系统性解决单位误识别、否定句漏判和拓扑误判等六类常见工程解析问题。

\textbf{(三) 构建了以 eZ-PLM 参考设计数据为上下文约束的 LLM 混合评分机制}。评分引擎首先对每个候选器件执行规则评分，包括参数匹配评分和供应链稳定性评分。其中，参数匹配评分采用动态约束归一化方法，根据实际存在的约束项数量计算均值，并综合电压覆盖、输出电压匹配、电流满足和温度范围适配等维度；供应链稳定性评分则根据库存水平和生命周期状态进行分段计算。当 LLM 接口可用且候选器件存在 eZ-PLM 关联参考设计时，系统调用 LLM 基于最多三条真实设计案例，对器件的应用场景适配度和工程设计成熟度进行独立评分，并按照参数匹配 40\%、供应链稳定性 10\%、应用适配度 25\% 和设计成熟度 25\% 的权重计算混合总分。当 LLM 不可用或缺少参考设计数据时，系统自动回退为基于规则的双维评分，即参数匹配 80\% 和供应链稳定性 20\%。评分完成后，系统通过核心型号提取函数合并同一系列的多封装变体。

第四，建立规则引擎事实验证与 LLM 工程叙述相分离的双层风险评估架构。规则层基于确定性逻辑对 13 类风险场景进行检测，包括无候选器件、仅单一推荐、推荐器件库存全部低于 100 件、存在停产生命周期器件、供应商集中度过高、推荐器件全部来自境外单一制造商导致的地缘政治风险、生命周期状态与库存信息全部缺失、封装数据全部缺失、需求要求车规但无认证器件，以及参数匹配平均分偏低等。每项检测结果均被记录为结构化的 \texttt{RiskItem}，包含风险类型、严重程度、风险描述、相关器件型号和缓解建议。LLM 层仅接收已验证的风险条目列表，并被限定为对已确认风险提供工程背景说明，不参与独立生成新的风险结论。报告中由 LLM 生成的叙述内容均附带明确的数据来源声明，从而实现风险事实与语言表达的分离，保障风险报告在工程评审场景下的严谨性与可追溯性。

第五，定义了一套可序列化为 JSON 的标准化中间表示（IR）数据模型体系，涵盖 PartIR（21 个字段，完整描述器件的制造商信息、电气参数、封装、认证等级、库存与生命周期状态）、ScoreBreakdown（各维度分数与评分模式）、TopologyIR（四子模型：拓扑节点图、外围元件参数表、热性能估算、Mermaid 框图与 Layout 规则）和 RiskIR（风险等级与 FMEA 量化字段），使得三类 Markdown 工程文档与结构化 TopologyIR JSON 能被同步生成，输出格式可直接服务于工程评审及 eZ-PLM 平台的数据互通。
